Issue #2094: Chat message visual order reversed

Root Cause: The user message header in TurnItem.tsx has class "sticky top-0 z-20 relative" (both sticky and relative). Inside the transform: translateY() container (from @tanstack/react-virtual), position: sticky does not work per CSS spec because transform creates a new containing block. The fallback behavior combined with z-index stacking contexts creates the visual reversal.

How to verify the bug in the browser

  1. Open a chat with multiple message exchanges (user question → assistant response → user question → assistant response)
  2. Scroll down in the chat viewport
  3. Observe the visual order: Does the assistant message appear above the user message that preceded it?
  4. Select text: Drag down from where a user message appears visually. If the selection highlights the assistant message above it instead, the bug is confirmed - the DOM order (correct) differs from the visual order (reversed).
  5. DevTools check:
    1. Inspect the user message element in TurnItem
    2. Check that it has both sticky and relative classes
    3. Check the computed position - should be sticky but if it's relative, that confirms a CSS cascade issue
    4. Check the stacking context: transform: translateY(Xpx) should create a stacking context on the tanstack virtualizer inner div
    5. Note that position: sticky does NOT work inside a transform container per CSS spec
    6. Check the z-index: user header is z-20, assistant block is z-0 - within the transform stacking context, z-20 paints on top

Root cause analysis

Primary issue: position: sticky inside a transform: translateY() container does not work per CSS specification. The @tanstack/react-virtual virtualizer wraps history items in a transform: translateY(startOffset) container (line 1201 of MessageList.tsx). This creates a new containing block, so the position: sticky; top: 0 on the user header does not stick to the scroll viewport as intended.

Secondary issue: The user header in TurnItem.tsx simultaneously applies sticky AND relative classes. While the compiled Tailwind v4 CSS correctly gives precedence to sticky (it's defined after relative), having both classes is unnecessary and could cause confusion.

Visual reversal mechanism: Inside the transform stacking context, the user header (z-20) paints on top of the assistant block (z-0). However, because sticky doesn't work inside transform, the user header stays in normal flow. Both elements are children of the same stacking context (the transform), and z-20 correctly paints above z-0. The visual reversal occurs when:

  1. In the streaming tail (no transform), sticky DOES work, so the user header behaves differently from the virtualized history
  2. Between virtualized turns, the broken sticky behavior combined with the bg-[var(--surface-background)] on the user header creates unexpected overlapping when consecutive turns are in view
  3. The z-index: 20 on the user header vs z-index: 0 on the assistant block means the user block from a PREVIOUS turn could paint on top of the assistant block from a LATER turn when scrolling
